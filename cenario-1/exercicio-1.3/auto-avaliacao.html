D1 Domínio Conceitual 3 / 3

O candidato demonstra compreensão real do que está sendo abstraído. Não usou LangChain — construiu diretamente sobre ChromaDB + sentence-transformers, o que prova que entende cada camada. Quem usa LangChain sem entender o substrato não consegue derivar por que _table_to_prose() é necessária, nem implementá-la corretamente.

  • ✓ Explica por que o all-MiniLM-L6-v2 embeddifica mal tabelas Markdown — pipe characters e repetição de células reduzem qualidade do vetor. Não apenas diz "a qualidade era baixa".
  • ✓ Re-ranking com penalidades derivadas de propriedades documentais (superseded −0.15, informal −0.10) mostra que retrieval não é apenas similaridade vetorial.
  • ✓ Lost-in-the-middle mitigation aplicada no montar_prompt() com justificativa explícita — chunk mais relevante posicionado por último.
  • ✓ Chunking semântico por heading H2/H3, não window fixo — demonstra que entende "chunk como unidade de contexto" (tópico central do exercício).
  • ✓ Dois perfis distintos: standard ≤400 palavras, table ≤1500 — tabelas nunca cortadas ao meio, preservando integridade semântica.
  • ✓ table_to_prose: prosa inserida no início do chunk (capturada pelo embedding), tabela original preservada ao final (lida pelo LLM). Dois consumidores, dois formatos.

A rubrica alerta: "Se usou LangChain sem entender o que está abstraído → D1 ≤ 2." O candidato tomou o caminho oposto.

O documento submetido nesta avaliação não contém nenhum prompt, completion, captura de iteração ou transcrição do Copilot. A regra não é discricionária.

O git log do repositório contém o commit feat: adiciona evidências de uso do Copilot CLI - Exercício 1.3, o que sugere que arquivos de evidência existem no repositório mas não foram incluídos neste entregável. Se for isso, a correção é colocar o conteúdo desse arquivo aqui.

Para atingir D2 = 3, a evidência precisa mostrar iteração real — não apenas "usei o Copilot":

  • → Pelo menos 3 prompts distintos com o contexto real passado ao Copilot durante a construção do pipeline
  • → Iteração demonstrada: v1 gerado → análise crítica do output → prompt refinado → v2 diferente (e por que melhorou)
  • → Ao menos um exemplo onde o Copilot errou e o que foi corrigido — prova que o output foi analisado criticamente, não aceito acriticamente
Observação Se as evidências existem no repositório e simplesmente não foram incluídas neste documento, D2 pode ir de 1 → 3 sem escrever uma linha de código. Este é o único item que impede nota máxima. A qualidade técnica do código já demonstra que o Copilot foi usado como amplificador, não como substituto — falta apenas a documentação disso.

O código é completo, funcional e específico ao NovaTech. Não é pseudocódigo — usa a API real do ChromaDB com PersistentClient, SentenceTransformer.encode(), e os resultados de teste exibem scores decimais precisos (0.8066, 0.7918) e contagem de chunks (39) que só existem mediante execução real.

  • ✓ 4 módulos completos com lógica real: config.py, ingestao.py, busca.py, prompt.py
  • ✓ 39 chunks indexados com metadados completos por documento (doc_id, status, version, source_type, chunk_type)
  • ✓ 5/5 testes com scores reais e comparação explícita ao gabarito do Anexo B — incluindo notas sobre posições e near-misses
  • ✓ 2 problemas reais identificados — ambos derivados dos testes, com mecanismo de causa descrito
  • ✓ Problema 1 resolvido com antes/depois quantificado: score 0.7362 → 0.7826, posição 6 → 3

Dois gaps de empacotamento que não cortam D3 mas que um avaliador vai notar:

  • ✗ Sem script integrador (main.py ou run_tests.py) — os 4 módulos existem mas não há demonstração explícita de como o pipeline completo é chamado end-to-end.
D4 Pensamento Crítico 3 / 3

Ponto mais forte do entregável. O candidato não apenas reporta resultados — explica causas, quantifica impacto das correções, e verifica ativamente a ausência de resultados errados (não só a presença dos corretos).

  • ✓ T3 — antes/depois quantificado com causa concreta: "pipe characters e repetição de células" como mecanismo, não vaga referência a "qualidade do embedding". Score +0.046, posição +3.
  • ✓ T1 — observação além do pass/fail: chunk de exceções (§3.2) recuperado em posição 7, não top-3. Propõe co-recuperação obrigatória como solução sistêmica, não ajuste de score.
  • ✓ T5 — verificação negativa: confirma que PROC-042-v1 (Sudeste=1.0) não apareceu no top-8. Não basta o resultado correto estar presente — é necessário que o resultado errado esteja ausente.
  • ✓ Problema 2 — mecanismo identificado: "vocabulário compartilhado ('cliente', 'prazo', 'carga') infla similaridade cross-domain." Proposta em duas frentes: metadado topic + BM25 híbrido.
  • ✓ Escala de produção referenciada: "Em produção com 1.250 documentos, separação seria naturalmente maior" — o candidato entende o limite do PoC sem minimizar o problema.
D5 Aplicabilidade ao Projeto 3 / 3

O pipeline está ancorado nos documentos reais da NovaTech, não em dados sintéticos. As decisões de design derivam diretamente das características conhecidas de cada fonte — não são configurações genéricas reutilizadas de tutorial.

  • ✓ Todos os 5 documentos NovaTech indexados com metadados corretos: POL-001, PROC-042 (v1 superseded), PROC-042-v2 (active), SLA-2024, FAQ-Atendimento
  • ✓ PROC-042 v1 marcado como superseded com penalidade de re-ranking dedicada — trata a armadilha de contradição entre versões como dado do domínio, não edge case genérico
  • ✓ Perguntas de teste específicas ao negócio NovaTech: carga perigosa (exclusão de escopo POL-001), multiplicador regional Sudeste, SLA Gold — não perguntas genéricas de "RAG demo"
  • ✓ System prompt com regras específicas para atendimento interno: escalar quando não encontrar, citar [Fonte: NOME, seção X.X], exceções antes da regra geral
  • ✓ FAQ-Atendimento classificado como informal com penalidade própria — reflete a distinção real entre política normativa e base de conhecimento informal da empresa
O que fazer antes de entregar
01
Incluir evidências do Copilot no documento +2 pontos · D2: 1 → 3
O commit feat: adiciona evidências de uso do Copilot CLI existe no repositório — se o arquivo não está aqui, copie-o agora. Se as evidências existem mas são superficiais (um único prompt, sem iteração visível), fortaleça: 3+ prompts com contexto real, ao menos uma iteração v1→v2 com análise crítica, um caso onde o Copilot errou e o que foi corrigido. Esta é a única ação que move o score total.
02
Adicionar requirements.txt alto impacto · 5 min
As versões já aparecem nos resultados: ChromaDB 0.5.23, sentence-transformers 3.3.1. Formalizar em requirements.txt torna o pipeline reproduzível e elimina qualquer dúvida sobre se o código "realmente roda". Cinco linhas. Um avaliador que tenta reproduzir e não consegue vai questionar D3.
03
Adicionar script integrador médio impacto · 15 min
Um main.py ou run_tests.py que chame ingest() → buscar(pergunta) → montar_prompt(pergunta, chunks) fecha o loop end-to-end. Não precisa ser elaborado — 20 linhas provam que os módulos se integram e que o pipeline é executável como unidade.
04
Incluir output real do terminal médio impacto · 5 min
Um bloco de saída com "Ingestão concluída: 39 chunks" e o resultado de uma busca com scores reais transforma números reportados em evidência verificável. Os scores já são específicos demais para serem inventados — um log do terminal os confirma definitivamente.
05
Justificar numericamente os parâmetros de chunking baixo impacto · 5 min
Os valores 400 palavras e 40 de overlap estão no config.py sem justificativa escrita. Uma linha no docstring de ingestao.py — como "400 palavras ≈ 520 tokens, dentro do orçamento estimado no Ex. 1.1" — conecta os exercícios da trilha e demonstra que o valor foi calculado, não arbitrado. Especialmente relevante porque o Exercício 1.1 exigia exatamente essa estimativa.
